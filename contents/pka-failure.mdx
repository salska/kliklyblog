My recent proof of concept for a Personal Knowledge Assistant (PKA) system was a masterclass in frustration, but it provided the exact "engineering reality check" I needed. I started with the populist view: that if I just crafted the perfect prompts, scripted the right skills, and layered on enough guardrails, I could build a fully scalable AI business capable of solving any problem.

The reality? It was a disaster. Despite all my "carefully crafted" artifacts, the system had no functional memory - it ignored existing procedures and generated a mountain of "documentation slop", creating slightly new variants of the same tasks every time. My tasklist never shrunk; it only grew. I couldn't coerce the orchestrator to reach a conclusion on anything.

My PKA failure was educational because it highlighted a fundamental truth: blind faith in LLM-based systems without deeper engineering insights will result in utter disappointment. Stop trying to build "smarter" agents and start building systems where failure is not an option. True reliability comes from something outside the LLM enforcing what happens next.

I spent weeks trying to "prompt my way" out of the mess. I thought better instructions would lead to better behavior. 

If a bad outcome is possible within your system's design - like an agent deciding to start a new task instead of finishing an old one - a probabilistic model will eventually produce it. No amount of detailed prompting or guardrails can change the fact that you cannot get deterministic behavior from a probabilistic source. 

My exploding tasklist was thanks to the "loop of reinterpretation". Instead of progressing toward completion, my PKA was continuously redefining the problem.

I had built a system that looked structured on the surface, but because the agent had the autonomy to decide "what happens next," it chose to reinvent the wheel every time it hit a snag. This is "quiet failure": the AI does something that looks right and consumes plenty of tokens, but it never actually gets anything done.

**The Engineering Pivot:** From Prompt-Driven to System-Driven
To save the project, I had to stop asking, “How do I prompt the agent better?” and start asking, “Why is the agent allowed to decide what happens next at all?”.

The new direction was clear. I had to move away from prompt-driven orchestration and toward system-driven orchestration. From a systems-thinking perspective, we have to stress-test the architecture itself. 

Enforce workflows in code, not prompts. Good solid state machines must manage the system transitions not the model. By making state explicit and authoritative, memory is no longer a vague "feeling" in a context window; it is a  rigid state managed outside the LLM. I used #n8n but #Camunda is also an option.

I am treating the LLM as a component, not a controller. The LLM is a tool for bounded decisions, not the manager of the entire process.

## The New Architecture: PKA Orchestrator

When I looked at the wreckage of my initial PKA proof of concept, I realised I hadn't built a system; I had built a sophisticated gambling machine that used tokens instead of chips. To fix it, I had to stop trusting the LLM to "manage" itself and instead implemented the PKA Orchestrator, a deterministic, state-driven architecture that treats AI agents as bounded components rather than autonomous drivers.

From my perspective as a systems analyst, here is how this new architecture specifically dismantled the points of failure that had previously defeated me:

1. Solving "No Memory" with File-Based State

My original system failed because it had no functional memory; it ignored artifacts and reinvented the wheel every time. I solved this by moving the "source of truth" out of the LLM's context window and onto the filesystem

**The Implementation:** I adopted a protocol where every task has its own dedicated directory (tasks/{task_id}/) containing specific JSON files like state.json and RESULT.json

**The Result:** Memory is no longer a "feeling" the agent has; it is a rigid, auditable history

The system uses a Knowledge Management (KM) repository where the Operating Model (PKA-OM) is injected into every task, ensuring the agent is always grounded in our standard operating procedures (SOPs)

2. Ending the "Loop of Reinterpretation" via n8n Orchestration

I was previously stuck in a "loop of reinterpretation" where the agent would just keep talking without doing. I broke this loop by implementing a "Hub-and-Spoke" architecture using n8n as a central state machine

**The Implementation:** The LLM does not decide the lifecycle of a task. Instead, n8n manages the flow (Start -> Delegate -> Poll -> Resolve -> Finish)

I use a script called poll_subtasks.py that waits in 45-second intervals to check if the agent has actually written a termination file like RESULT.json

**The Result:** The system is now deterministic. If the agent doesn't produce a valid result or a specific question, it doesn't move forward. We’ve replaced "vague interpretation" with "deterministic routing"

3. Controlling the Tasklist with Orka (The Master Orchestrator)

The issue of my ever-growing, never-shrinking tasklist was solved by centralizing "task birth." I introduced Orka, a Master Orchestrator responsible for Quality Assurance and blocker resolution.

**The Implementation:** A task only terminates when Orka writes an APPROVED.json file

If there is a gap—like a missing security policy—Orka either resolves it using system knowledge (resolve_blocker.py) or escalates it to me via a macOS notification (ESCALATE.json)

**The Result:** This created "Convergence via Approval"

The task list only grows if Orka identifies a specific, missing piece (e.g., "You found a bug but didn't update the SOP") and explicitly spawns a new, linked sub-task

4. Replacing "Quiet Failure" with "Loud" Escalation

In my old setup, if an agent got stuck, it just hallucinated a way out or stalled quietly. Now, the system is designed to fail "loudly" and helpfully.

**The Implementation:** When an agent lacks the deterministic rules for a task, it is programmed to write a QUESTION.json
. Orka then attempts to unblock it by writing an ANSWER.json
. If the decision is strategic (e.g., "Should we migrate to AWS or Azure?"), the system pauses and waits for human input

**The Result:** By enforcing these role boundaries, I’ve ensured that the agent never has to "guess" when it's out of its depth

By putting on my engineering head and building this framework, I’ve moved from a system of "blind faith" to one of "systemic control." I no longer ask the agent what to do; I have a state machine that tells the agent what its job is, and a Master Orchestrator that ensures the job actually gets done.

Life is good & getting better. The system is on the path to  stability, and I can finally see the path forward. The PKA is no longer a black box that might explode at any moment; it's a reliable component in a well-engineered system.
